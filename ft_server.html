hkamiya Aug 04, 2020 11:03 AM
@yiwasa
>これは自己証明書をつかっている場合普通のことなのでしょうか？
Yesです。

理解するためには、ブラウザにおける証明書とはなにかを理解する必要があります。

SSLは2つの役割があります。
1つは通信を暗号化すること。
もう1つは接続先が「正しい」サイトなのかを接続元が確認できること。
1つ目は感覚的にわかると思いますが、2つ目は感覚的ではないかもしれないです。

https://profile.intra.42.fr/に接続して表示されたWebページは本当に42のHPなのでしょうか。
もしかしたら、42のHPではなく、悪意のある第三者が42.frを乗っ取って表示しているのかもしれません。
この状況を防ぐため、接続しようとしているサイトが本物であることを証明する仕組みがあります。
それが証明書です。

証明書の仕組みを理解するには公開鍵暗号の仕組みを理解する必要があります。
通信の暗号化をするためには、暗号鍵と公開鍵を作ります。
公開鍵で暗号化したものは暗号鍵で復号できます。
通信を暗号化する場合は、まず送信元が公開鍵を相手に渡し(もしくは相手が公開鍵をGETし)、送信元が暗号鍵でデータを暗号化して送信し、送信先は公開鍵でデータを復号化します。

面白いことに、鍵の使い方を逆にしてもよいです。
暗号鍵で暗号化したものを公開鍵で復号することができます。
この仕組みは、改ざんの検知、送信元の確認に使われます。
送信元は自分が送るデータに暗号鍵で鍵をかけます。データを受け取った相手は公開鍵を使って鍵を解除します。
公開鍵で鍵を解除できるのは、その鍵と対になる暗号鍵で鍵をかけたデータだけです。
こうして、改ざんがされていないことと送信元が公開鍵と対になる暗号鍵の持ち主であることが確認できます。 (edited)

@yiwasa
証明書の説明にうつります。
SSLプロトコルにおける証明書とは、データの送受に使われる公開鍵が送信元のドメインの持ち主であることを第三者が証明したデータです。
42.frを例に説明します。
42.frでSSLを使うために、公開鍵と暗号鍵を作成します。
この鍵が42.frの持ち主であることを証明するために第三者機関（いろいろあります）に作った公開鍵にいくつかの情報を追加しておくります。
第三者機関は、公開鍵+いくつかの情報に第三者機関の暗号鍵で鍵をかけ、42.frの持ち主に送り返します。
送り返したデータが証明書です。

証明書の使い方にうつります。
42.frにクライアントが接続すると、42.frはクライアントに証明書を送ります。
クライアントは、証明書が正しいのかを確認するため第三者機関の公開鍵で証明書の鍵を解除します。
（第三者機関の公開鍵はブラウザに組み込まれてます。）
解除できれば、その証明書は正しいので、その証明書に含まれる42.frの公開鍵で暗号化の通信を開始します。
このようにして、SSL通信が始まる前に、そもそも接続先が正しいのかを確認してます。

やっと、自己証明書の説明ができます。
自己証明書は第三者機関の証明を受けていないので、クライアントにはその証明書が正しいかどうかはわかりません。
具体的には、その証明書を使うと、データは暗号化されるが、そもそも通信相手が正しくない可能性があります。
なので、その証明書を使って通信をしようとすると「その証明書を使って通信してもいいのか？」と警告を発します。

以上から
>これは自己証明書をつかっている場合普通のことなのでしょうか？
の回答はYesです。

@rnitta さんの考える「環境」の要素がなんなのか?で回答が違ってしまいます。※環境では、ざっくりしすぎています。
dockerは、Linuxの枯れた技術である、cgroups、namespacesという技術を使っています。
https://www.itbook.info/network/docker06.html
cgroups、namespacesを使ってコンテナを作ると、何をホストOSと共有し、何をホストOSと分離するのかを明らかにする必要があります。
環境の要素のうちAは、ホストOSと共有するのであれば(例えば確か/proc配下は共有します))、Aはモロにコンテナに影響を与えます。
環境の要素のうちBは、ホストOSと共有しないのであれば(ほとんどの要素は分離されます)、BはホストOSおよび他のコンテナに影響を与えません。
環境のほとんどの要素はBに属するので、開発環境でDcokerを使う分には、環境による差はVMと遜色ないくらいほとんど0といってよいです、たぶん。

可変にしたい設定がある場合は、docker runする際に環境変数として渡してあげて、コンテナの起動スクリプトの中で受け取った環境変数にしたがって設定を書き換えて起動するのが一般的です。
https://docs.docker.jp/engine/reference/commandline/run.html#e-env-env-file
MySQLの公式のイメージ( https://hub.docker.com/_/mysql/ )では、MYSQL_ROOT_PASSWORDL、MYSQL_DATABASEのような環境変数があります。
※設定できる値は「Environment Variables」を参照ください。
MySQLの公式のイメージを使うと、docker runする場合に渡された、MYSQL_ROOT_PASSWORD、MYSQL_DATABASEをMySQLの設定に反映します。

MySQLの公式のイメージのDcokerfileは
https://github.com/docker-library/mysql
にあります。
MYSQL_ROOT_PASSWORDがどのように設定されているか…というと、MySQL8.0の場合、
https://github.com/docker-library/mysql/blob/master/8.0/Dockerfile
の中の、82行目のENTRYPOINT ["docker-entrypoint.sh"]で設定されていて、docker-entrypoint.shは
https://github.com/docker-library/mysql/blob/master/8.0/docker-entrypoint.sh
にありまして、236行目で

CREATE USER 'root'@'${MYSQL_ROOT_HOST}' IDENTIFIED BY '${MYSQL_ROOT_PASSWORD}' ;

とあるので、docker runする際に、環境変数MYSQL_ROOT_PASSWORDで設定した値が反映されていることがわかります。

同様に、autoindexの設定を環境変数で渡してあげて、docker runする際に…つまりnginxを立ち上げる前に設定ファイルを書き換えてあげて、nginxをたちあげてあげれば、フレキシブルにautoindexをonにしたりoffにしたりすることができます。 (edited)

原因が分かりました。
結論から書くと、init_docker.shの3行目のmkdir /var/run/mysqldを削除することで所望の動作をすることを確認しました。

以下、検証結果を記載します。

コンテナに入りinit_docker.shを実行するも30秒後にFailすることを確認。
? ft_server_debug git:(main) docker run -it -p 8080:80 test bash
root@6d2d68617f16:/# chmod 777 init_docker.sh
root@6d2d68617f16:/# ./init_docker.sh
[....] Starting MariaDB database server: mysqld . . . . . . .


service mysql startを直接実行するもFailすることを確認
root@6d2d68617f16:/# service mysql start
[....] Starting MariaDB database server: mysqld . . . . . . . . . .^C
service mysql startは/etc/init.d/mysql startを実行しているのと同等なので、/etc/init.d/mysqlの内容を確認
[/etc/init.d/mysql]
case "${1:-''}" in

'start')
sanity_checks;
# Start daemon
log_daemon_msg "Starting MariaDB database server" "mysqld"
if mysqld_status check_alive nowarn; then
log_progress_msg "already running"
log_end_msg 0
else
# Could be removed during boot
test -e /var/run/mysqld || install -m 755 -o mysql -g root -d /var/run/mysqld　←　(1)

# Start MariaDB!
/usr/bin/mysqld_safe "${@:2}" 2>&1 >/dev/null | $ERR_LOGGER &　←　(2)

for i in $(seq 1 "${MYSQLD_STARTUP_TIMEOUT:-30}"); do　←　(3)
sleep 1
if mysqld_status check_alive nowarn ; then break; fi
log_progress_msg "."
done
if mysqld_status check_alive warn; then　←　(4)
log_end_msg 0
# Now start mysqlcheck or whatever the admin wants.
output=$(/etc/mysql/debian-start)
if [ -n "$output" ]; then
log_action_msg "$output"
fi
else
log_end_msg 1
log_failure_msg "Please take a look at the syslog"
fi
fi
;;
(2)の部分で/usr/bin/mysqld_safeを実行していて、(3)の部分でmysqldが立ち上がったかどうかを30秒チェックしていて、(4)で立ち上がった場合とそうでない場合で分岐している。
というわけで、(1)のコマンドの内容を見ると、
test -e /var/run/mysqld

/var/run/mysqldがあるかをチェックしていて、/var/run/mysqldがなかったら
install -m 755 -o mysql -g root -d /var/run/mysqld

を実行している。
このコマンドは、ディレクトリ/var/run/mysqldを権限755で所有者がmysql、所有グループはrootとして作る。
このコマンドを実行すると
# ls -l /var/run/
total 16
drwxr-x--- 2 Debian-exim Debian-exim 4096 Dec 16 12:42 exim4
drwxrwxrwt 2 root root 4096 Sep 8 07:00 lock
drwxr-xr-x 2 mysql root 4096 Dec 16 12:53 mysqld ←
drwxr-xr-x 2 www-data www-data 4096 Dec 16 12:44 php
-rw-rw-r-- 1 root utmp 0 Sep 8 07:00 utmp

となる。
この後の(2)で実行される/usr/bin/mysqld_safeはセキュリティの都合でroot権限では実行できない。
なので、mysqld_safeを実行するホームディレクトリ(だと思う…)の所有者はrootではなく、mysqlである必要がある。

ところで、init_docker.shの中で、service mysql startを実行する前に、「mkdir /var/run/mysqld」が実行されている。
この結果、
# ls -l /var/run/
total 16
drwxr-x--- 2 Debian-exim Debian-exim 4096 Dec 16 12:42 exim4
drwxrwxrwt 2 root root 4096 Sep 8 07:00 lock
drwxr-xr-x 2 root root 4096 Dec 16 12:53 mysqld ←
drwxr-xr-x 2 www-data www-data 4096 Dec 16 12:44 php
-rw-rw-r-- 1 root utmp 0 Sep 8 07:00 utmp

となる。mysqldの所有者はrootになる。
この状態で、service mysql startを実行すると、
test -e /var/run/mysqld || install -m 755 -o mysql -g root -d /var/run/mysqld

のコマンドは、test -e /var/run/mysqldの部分の返り値が1となるので、後続のinstall -m 755 -o mysql -g root -d /var/run/mysqldは実行されない。
そのため、/var/run/mysqldの所有者はrootのままとなる。
この状態では、/usr/bin/mysqld_safeの実行は失敗します。

以上が検証結果です。

自分はmysqld_safeがmysqlユーザーでないと起動できないことをたまたま知っていたので「たぶん、権限が間違ってるんだろうなあ…」と目星をつけてスクリプトを見ました。

default_serverパラメータがない場合には、最初にIPアドレスとポート番号の条件に一致したlistenディレクティブを持つserverディレクティブがデフォルトサーバになります。本記事の最初に述べたように複数のバーチャルサーバ毎の設定ファイルがある場合には、読み込まれる順番は不定であるため、どのバーチャルサーバの設定が最初に読み込まれるかはわかりません。そのため、明示的にdefault_serverパラメータを付けた方がよいでしょう。

http://nginx.org/en/docs/http/ngx_http_core_module.html#listen
に
> The default_server parameter, if present, will cause the server to become the default server for the specified address:port pair. If none of the directives have the default_server parameter then the first server with the address:port pair will be the default server for this pair.
と記載があります。
addressとportの組み合わせが同じserverディレクティブが複数あった場合、どのsever_nameにもマッチしなかった場合に読み込むserverディレクティブを決定するパラメーターがdefault_serverです。


(1)
server {
listen 80 defult_server;
.......
}
(2)
sever {
listen 443 ssl defult_server;
.......
}

の場合、(1)と(2)の設定はポートが違うので、80番ポートで待ち受けるデフォルトのサーバーの設定が(1)、443番ポートで待ち受けるデフォルトのサーバーが(2)になります。

ええと…defult_serverは以下のように使います。

(1)
server {
listen 80 default_server;
.......
}

(2)
server {
listen 80;
server_name example.com example.net
.......
}

な設定で、このサーバーのIPアドレスが192.168.0.100で、example.com example.net example.orgを名前解決すると、192.168.0.100な場合、

(1)に接続される
http://192.168.0.100/
http://example.org/

(2)に接続される
http://example.com/
http://example.net/

Wordpress初回起動時に言語設定が出ずにサイト設定が出てしまう現象があったので原因と解決方法を調査しました。
（inception課題をやっているうちに調べた内容ですが、ft_serverやっている時も起きたのでこちらに）

原因：
初回起動ページのinstall.phpは中で画面が分かれていて１ページ目（install.php?step=0）が言語選択画面。
言語選択画面を出すか、次のサイト設定（install.php?step=1）を出すかは、wp-contentの実行権限を調べる等した結果による。
調べる内容は色々ありそうだったけど、私がfalseにされてしまっていたのは、
fopen(/wordpressまでのパス/wp-content/temp-write-test-xxxxxxx, 'w');でファイル作成して作成したファイルの権限と/wordpressまでのパス/wp-admin/includes/file.phpの権限が同じかどうか調べる項目。

解決方法：
Wordpress(php-fpm)を実行しているユーザとwordpressのファイルやディレクトリのユーザを合わせる。

私の場合はファイルやディレクトリのユーザや権限は良かったんですが、www.confのuserとgroupの設定をしていませんでした。
